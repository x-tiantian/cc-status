# cc-status 项目需求文档(PRD)

> 版本:v1.0　日期:2026-07-09　状态:待评审
> 关联文档:[可行性分析.md](./可行性分析.md)

---

## 1. 背景与目标

### 1.1 背景
开发者在使用 Claude CLI(Claude Code)时,常常同时运行**多个实例**(对应多个项目/仓库),或在等待 Claude 工作时切换去做别的事情。Claude CLI 在运行中会出现需要**人工介入**的时刻——例如等待用户输入下一步指令、请求执行敏感操作的授权。若开发者没有盯着某个终端窗口,这些时刻很容易被错过,导致 Claude 长时间空等。

### 1.2 目标
提供一个常驻托盘旁的**极简可视化状态面板**,用"红绿灯"直观呈现每个 Claude CLI 实例的当前状态,使开发者**无需盯着终端**即可在余光中掌握全部实例的运行情况,并在需要对话或授权时**第一时间介入**。

### 1.3 非目标(本期不做)
- 不做 Claude CLI 的启动/管理/进程控制,只做**状态展示**。
- 不做历史记录/日志分析/统计报表。
- 不做 macOS / Linux 版本(仅 Windows 10+ x86-64)。
- 不做点击灯直接跳转/唤起对应终端(可作为未来增强)。

## 2. 名词定义

| 术语 | 含义 |
|------|------|
| **项目 / Project** | 一个 Claude CLI 实例的标识,通常等于工作目录名或仓库名。cc-status 以 `project` 字段区分不同的灯。 |
| **灯 / Light** | 代表一个项目当前状态的圆形指示器,颜色随状态变化。 |
| **状态 / State** | 项目所处的工作阶段(见 §5.1 状态机),决定灯的颜色与动画。 |
| **心跳 / Heartbeat** | 来源每次推送即视为一次心跳,用于判断项目是否仍然存活。 |
| **面板 / Panel** | 承载全部灯的、锚定在托盘左侧的自绘置顶窗口。 |
| **来源 / Reporter** | 发送状态推送的一方,本期为 Claude Code 的 Hook 脚本。 |

## 3. 用户画像与核心场景

**用户画像**:同时管理多个仓库、重度使用 Claude CLI 的开发工程师;习惯多窗口/多任务;希望"少切换、被动感知"。

**核心场景**:
1. **多实例并行**:开发者开了 3 个 Claude CLI 分别跑不同仓库,去写文档;某个实例进入"等待授权"→ 对应灯变橙闪 → 开发者瞥见后切回该终端处理。
2. **超 3 个分屏**:开发者同时跑了 5 个实例;面板每屏显示 3 个,每隔数秒上下翻页轮播,保证都能被看到。
3. **查看详情**:开发者不确定某个黄灯具体在等什么,鼠标悬停 → 弹出提示显示"等待输入 · <Claude 的消息>"。
4. **调整配置**:开发者想换端口或开启开机自启 → 右键面板 → 设置窗中修改并保存。
5. **跨机器监控**:开发者在 A 机跑 Claude CLI、在 B 机办公;A 机的 hook 将状态推送到 B 机上 cc-status 监听的局域网地址(如 `192.168.1.20:9898`),B 机面板集中展示多台机器上所有实例的状态。

## 4. 功能需求

### FR-1 状态监听服务
- FR-1.1 程序启动后在配置的 `IP:端口`(默认 `127.0.0.1:9898`)监听 HTTP。
- FR-1.2 提供两个接收端点:`POST /hook` 接收 Claude Code **原始 hook JSON**(主路径,服务端内部映射状态,见 §5.3);`POST /status` 接收 cc-status **自定义精简协议**(§5.2,供自定义/跨机器脚本上报)。解析成功后更新对应项目状态并刷新面板。
- FR-1.3 对无法解析/字段缺失的请求返回 4xx 且不崩溃;对合法请求返回 `200`。
- FR-1.4 **支持跨机器监控**:监听 IP 可配置为局域网地址(如 `0.0.0.0` 或本机某网卡 IP),使其他机器上的 Claude CLI hook 能把状态推送过来。当配置为**非回环地址**时,在设置窗给出安全提示(见 FR-1.6),并建议启用 token 校验。
- FR-1.5 端口被占用时,启动不崩溃,通过托盘图标气泡/设置窗提示,并允许改端口后重试。
- FR-1.6 **安全**:默认绑定 `127.0.0.1`;当监听非回环地址时,提示"该端口将对局域网开放";支持配置共享 `token`,请求需携带匹配的 `X-CC-Token` 头,否则返回 401。跨机器场景强烈建议启用 token。

### FR-2 项目与状态管理
- FR-2.1 以 `project` 字段为键维护项目集合;首次出现即新增一个灯。
- FR-2.2 每次推送刷新该项目的状态、消息与最后心跳时间。
- FR-2.3 **心跳超时**:超过 `offline_timeout`(默认 60s)无推送 → 状态置为 `offline`(灰)。
- FR-2.4 **自动移除**:进入 `offline` 后再持续 `remove_timeout`(默认 300s)无推送 → 移除该灯。
- FR-2.5 超时阈值可在设置中调整(本期可先固定默认值,设置项作为增强)。

### FR-3 面板显示(托盘左侧)
- FR-3.1 面板为无边框、置顶、不抢焦点、不在任务栏/Alt-Tab 出现的自绘窗口。
- FR-3.2 面板视觉上锚定于系统托盘**左侧**,垂直居中于任务栏;跟随任务栏位置(上/下/左/右)、DPI、分辨率、资源管理器重启而重定位。
- FR-3.3 每个项目渲染为一个圆形灯;颜色与动画由状态决定(见 §5.1)。
- FR-3.4 **最多同屏 3 个灯**;超过 3 个时分屏,每屏 3 个,按固定间隔(默认 4s)**上下翻页轮播**,循环显示所有分屏。
- FR-3.5 翻页时有简单过渡(如上下滑动/淡入淡出),并有轻微的"当前第 X/共 Y 屏"暗示(如小圆点指示,增强项)。
- FR-3.6 定位失败时降级为屏幕右下角悬浮显示。

### FR-4 悬停提示(Tooltip)
- FR-4.1 鼠标悬停在**某个具体灯**上,弹出提示,内容为:`<状态中文名> · <该项目最新 message>`,并显示项目名。
- FR-4.2 提示需命中测试到具体灯(多个灯时区分);移开即消失。
- FR-4.3 message 过长时截断并省略号;为空时仅显示状态名。

### FR-5 右键设置菜单/窗口
- FR-5.1 在面板任意处**右键**弹出菜单/设置入口(托盘图标右键亦提供同等入口作为兜底)。
- FR-5.2 设置窗可修改:**监听 IP**、**监听端口**(默认 `127.0.0.1:9898`)。
- FR-5.3 设置窗可切换:**开机启动**(默认**关闭**)。
- FR-5.4 (增强)可配置:超时阈值、轮播间隔、token、主题。
- FR-5.5 保存后立即生效:IP/端口变更触发监听服务重启;开机自启写入/移除 `HKCU\...\Run`。
- FR-5.6 设置窗提供"退出程序"入口。

### FR-6 配置持久化
- FR-6.1 配置存于 `%APPDATA%\cc-status\config.json`,首次运行生成默认配置。
- FR-6.2 配置项:`listen_ip`、`listen_port`、`autostart`、`offline_timeout`、`remove_timeout`、`carousel_interval`、`token`(可选)、`theme`(可选)。
- FR-6.3 配置文件损坏时回退默认值并重建,不崩溃。

### FR-7 开机自启
- FR-7.1 通过 `HKCU\Software\Microsoft\Windows\CurrentVersion\Run` 写入当前 exe 路径实现用户级自启,无需管理员。
- FR-7.2 关闭时移除该注册表项。
- FR-7.3 exe 移动位置后再次保存设置可刷新路径。

### FR-8 托盘图标(入口与兜底)
- FR-8.1 提供标准系统托盘图标,承载右键菜单(设置、退出)。
- FR-8.2 关键事件(如端口占用、定位降级)通过托盘气泡提示。
- FR-8.3 托盘图标保证即使面板定位失败,用户仍可访问设置与退出。

## 5. 状态机与协议

### 5.1 状态定义与视觉映射

| 状态 `state` | 含义 | 灯色 | 动画 | 是否需介入 |
|--------------|------|------|------|:---:|
| `idle` | 空闲 / 任务完成待命 | 🟢 绿 | 常亮 | 否 |
| `working` | 正在思考 / 执行中 | 🔵 蓝 | 呼吸 | 否 |
| `waiting_input` | 等待用户输入/对话 | 🟡 黄 | 缓闪/呼吸 | **是** |
| `waiting_permission` | 等待用户授权 | 🟠 橙 | 快闪 | **是** |
| `error` | 出错 | 🔴 红 | 常亮 | **是** |
| `offline` | 超时无心跳 | ⚪ 灰 | 常亮 | 否 |

> 需介入的状态(黄/橙/红)在视觉上更醒目(闪烁),便于余光捕捉。颜色为初步方案,允许开发期微调。

### 5.2 推送协议

**请求**:`POST /status`　`Content-Type: application/json`

```jsonc
{
  "project": "my-repo",          // 必填。项目标识,通常取工作目录名
  "state":   "waiting_input",    // 必填。见 §5.1 状态枚举
  "message": "是否允许执行 git push?", // 可选。悬停提示显示的内容
  "host":    "DEV-PC-A",         // 可选。来源主机名;跨机器时用于区分同名项目
  "session": "abc123",           // 可选。会话/实例 id;同一项目多开时区分不同灯
  "ts":      1752000000          // 可选。Unix 秒;缺省则服务端取当前时间
}
```

> **灯的唯一键**:服务端以 `(host, project, session)` 组合作为一个灯的唯一标识(缺省字段视为空)。单机单开时仅 `project` 即可;跨机器或同项目多开时,用 `host` / `session` 避免不同实例被合并到同一个灯。悬停提示会显示 `host/project`。

**响应**:
- `200 OK`:接受并更新。
- `400 Bad Request`:JSON 非法或缺少必填字段。
- `401 Unauthorized`:启用 token 且校验失败。

**兼容性约定**:未知的额外字段被忽略;未知的 `state` 值归一化为 `working`(或按配置忽略),保证前向兼容。

### 5.3 与 Claude Code Hooks 的集成(已调研确认)

> 本节结论已通过官方文档([code.claude.com/docs/en/hooks](https://code.claude.com/docs/en/hooks))与多来源交叉验证。

#### 5.3.1 采用原生 `http` 类型 hook,零脚本

Claude Code 的 hook 支持 **`type: "http"`**:事件触发时,Claude Code 会把该事件的 JSON **原样作为 POST body** 发送到配置的 URL(`Content-Type: application/json`)。因此:

- **无需 curl、PowerShell 或任何包装脚本**,Windows 上开箱即用,不受 shell 环境影响。
- HTTP hook 的连接失败/超时/非 2xx 均为**非阻塞错误**——即使 cc-status 未运行,Claude 也照常工作,不受影响。
- cc-status 只需对每个请求返回 `200 OK` 即可,不参与 Claude 的任何决策。

#### 5.3.2 双端点设计

cc-status 监听服务提供两个端点:

| 端点 | 数据格式 | 用途 |
|------|----------|------|
| `POST /hook` | Claude Code **原始 hook JSON** | 主路径。用户在 `settings.json` 配置 http hook 指向此端点,cc-status 服务端按 `hook_event_name` 内部映射为状态 |
| `POST /status` | cc-status **自定义精简协议**(§5.2) | 备用/通用路径。供自定义来源、非 hook 场景、或跨机器脚本手动上报 |

服务端从原始 hook JSON 提取:项目名 = `cwd` 的目录名;实例 id = `session_id`;主机 = 请求来源 IP(hook JSON 不含主机名,由连接地址补充);状态 = 由下表映射。

#### 5.3.3 事件 → 状态映射(仅需 7 个事件)

| Claude Code Hook 事件 | matcher | 关键字段 | → `state` |
|------|------|------|------|
| `SessionStart` | — | `source` | `idle`(注册新实例) |
| `UserPromptSubmit` | — | `prompt` | `working` |
| `PreToolUse` | `""`(全部) | `tool_name`,`tool_input` | `working`(刷新心跳) |
| `Notification` | `permission_prompt` | `message`,`notification_type` | `waiting_permission` |
| `Notification` | `idle_prompt` | `message` | `waiting_input` |
| `Stop` | — | `stop_hook_active` | `idle` |
| `StopFailure` | — | (`rate_limit`/`overloaded`/`billing_error`) | `error` |
| `SessionEnd` | — | — | `offline` → 移除 |

> `Notification` 用 matcher `permission_prompt|idle_prompt` 一条配置覆盖两种情况;可选加 `PermissionRequest` 事件获得更丰富的 `tool_name` 信息作为 `waiting_permission` 的增强来源。

**共有字段**(所有事件均含):`session_id`、`transcript_path`、`cwd`、`hook_event_name`,多数含 `permission_mode`。

#### 5.3.4 用户配置(随发布提供,复制即用)

在 `~/.claude/settings.json`(全局)加入,所有项目自动上报;统一指向 `/hook`,服务端按事件分发:

```jsonc
{
  "hooks": {
    "SessionStart":     [{ "hooks": [{ "type": "http", "url": "http://127.0.0.1:9898/hook", "timeout": 5 }] }],
    "UserPromptSubmit": [{ "hooks": [{ "type": "http", "url": "http://127.0.0.1:9898/hook", "timeout": 5 }] }],
    "PreToolUse":       [{ "matcher": "", "hooks": [{ "type": "http", "url": "http://127.0.0.1:9898/hook", "timeout": 5 }] }],
    "Notification":     [{ "matcher": "permission_prompt|idle_prompt", "hooks": [{ "type": "http", "url": "http://127.0.0.1:9898/hook", "timeout": 5 }] }],
    "Stop":             [{ "hooks": [{ "type": "http", "url": "http://127.0.0.1:9898/hook", "timeout": 5 }] }],
    "StopFailure":      [{ "hooks": [{ "type": "http", "url": "http://127.0.0.1:9898/hook", "timeout": 5 }] }],
    "SessionEnd":       [{ "hooks": [{ "type": "http", "url": "http://127.0.0.1:9898/hook", "timeout": 5 }] }]
  }
}
```

> 跨机器:把 URL 中的 `127.0.0.1` 换成 cc-status 所在机器的局域网 IP(如 `http://192.168.1.20:9898/hook`),并配合 token(§FR-1.6)。若某些环境 http hook 不可用,发布包附带 command 版兜底(`curl -s -X POST .../hook -H 'Content-Type: application/json' -d @-`)。

#### 5.3.5 状态机

```
 SessionStart ─▶ idle ◀────────────── Stop
                 │                      ▲
        UserPromptSubmit                │
                 ▼                      │
              working ──PreToolUse──┐   │ (心跳刷新)
                 │  ◀───────────────┘   │
     ┌───────────┼──────────────┐       │
     ▼           ▼              ▼        │
 Notification  Notification  StopFailure │
 permission_    idle_prompt      │       │
  prompt          │              ▼       │
     ▼            ▼           error      │
 waiting_     waiting_input             │
 permission       │                     │
     └─────────UserPromptSubmit─────────┘
 SessionEnd ─▶ offline ─▶(超时)移除
```

## 7. 交互与视觉细节

- **灯尺寸**:直径约 12–16px(随 DPI 缩放),灯间距约 6–8px;整体面板高度贴合任务栏高度(约 40px)。
- **配色**:采用饱和度适中、深浅背景下均清晰的颜色;灯带柔和外发光以提升可辨识度。
- **呼吸/闪烁**:用低频定时器驱动(如 30–60ms 一帧),仅在有动画状态存在时刷新,避免持续重绘。
- **轮播**:默认每 4s 翻一屏;鼠标悬停面板时**暂停轮播**,便于查看。
- **Tooltip**:自绘,深色半透明背景,白字;跟随灯位置弹出。
- **设置窗**:标准 Win32 对话框风格;字段含 IP、端口、开机自启复选框、保存/取消、退出程序。

## 8. 验收标准(关键)

1. 双击 exe,无需安装即在托盘左侧出现面板;托盘出现图标。
2. 用 curl 向 `127.0.0.1:9898/status` POST 一条 `waiting_permission`,对应项目灯在 1s 内变橙闪;悬停显示消息。
3. 推送 5 个不同 `project`,面板每屏 3 个并每 4s 上下翻页,5 个都能轮到。
4. 停止对某项目推送:60s 后转灰;再 300s 后灯消失。
5. 右键面板/托盘 → 设置 → 改端口保存 → 新端口生效、旧端口失效;勾选开机自启 → 注册表 Run 出现对应项,重启后自动运行;默认未勾选。
6. 任务栏切换到屏幕顶部/侧边、切换 DPI、重启资源管理器后,面板都能正确重新定位(或降级右下角)。
7. 全程 CPU 空闲接近 0、内存 < 30MB、长时间运行不崩溃。
