# cc-status 可行性分析报告

> 版本:v1.0　日期:2026-07-09　状态:已评审通过

## 1. 项目概述

`cc-status` 是一个运行于 Windows 的**轻量级状态监控托盘程序**,用 Rust 开发。它监听一个固定端口,接收来自一个或多个 Claude CLI 实例的状态推送,并在任务栏系统托盘左侧以"红绿灯"样式实时展示每个项目(Claude CLI 实例)的工作状态,帮助开发者在同时运行多个 Claude CLI、或一边写代码一边处理其他工作时,及时发现"需要对话/需要授权"等需要人工介入的时机。

## 2. 需求可行性逐条评估

| # | 需求 | 可行性 | 技术方案 | 风险 |
|---|------|--------|----------|------|

| 3 | 托盘区**左侧**显示工作状态 | ⚠️ 有条件可行 | 自定义无边框置顶窗口,锚定到托盘左侧(见 §3) | 中:定位需适配多任务栏位置/DPI |
| 4 | 多个 Claude CLI 状态(按项目区分) | ✅ 可行 | 内存中维护 `project_id -> 状态` 映射 | 无 |

**总体结论:项目整体可行。** 唯一需要特别设计的是需求 3「托盘左侧定位」,已有成熟的 Win32 定位方案,风险可控。

## 3. 关键技术难点:如何"在托盘左侧显示"

### 3.1 结论:不使用已废弃的 Deskband

微软官方 UX 指南已将 **Deskband(桌面工具栏,可真正嵌入托盘栏内部)标记为"不推荐(Not recommended)"** 并逐步废弃。Deskband 需要注册为 COM 进程内服务器(需安装、需 regsvr32),与"免安装双击即用"的需求直接冲突。因此**不采用**。

### 3.2 采用方案:锚定的置顶自绘窗口

创建一个自有窗口,使其视觉上紧贴托盘区左侧:

- **窗口样式**:`WS_POPUP`(无边框)+ `WS_EX_TOPMOST`(置顶)+ `WS_EX_TOOLWINDOW`(不出现在任务栏和 Alt-Tab)+ `WS_EX_NOACTIVATE`(点击不抢焦点)。可选 `WS_EX_LAYERED` 做透明/圆角。
- **定位算法**:
  1. `FindWindow("Shell_TrayWnd")` 找到任务栏,再找子窗口 `TrayNotifyWnd`(通知区域)。
  2. 用 `GetWindowRect` 取通知区域左边界,把本窗口右边界对齐到该左边界,垂直居中于任务栏。
  3. 处理任务栏在上/下/左/右四种停靠位置(`SHAppBarMessage` / `ABM_GETTASKBARPOS`)与多显示器。
- **跟随更新**:监听 `TaskbarCreated`(资源管理器重启)、`WM_DPICHANGED`、`WM_SETTINGCHANGE`(任务栏移动/分辨率变化),重新计算位置;并用低频定时器兜底纠偏。
- **每帧绘制**:Direct2D(首选,抗锯齿+动画流畅)或 GDI(兜底)绘制若干圆形"灯",支持颜色切换与呼吸动画。

### 3.3 备选/降级方案

若某些系统(如自定义 shell、异形任务栏)锚定失败,降级为**屏幕右下角固定悬浮**;同时始终提供一个标准 `tray-icon` 托盘图标作为入口(承载右键菜单、双击打开设置),保证核心功能不因定位失败而不可用。

## 4. 技术栈选型

| 领域 | 选型 | 说明 |
|------|------|------|
| 语言/工具链 | Rust 1.88 + MSVC 工具链 | 目标 `x86_64-pc-windows-msvc`(**本机已就绪**) |
| Win32 绑定 | [`windows`](https://crates.io/crates/windows)(windows-rs 官方) | 微软官方维护,覆盖 Win32/Direct2D 全 API |
| 托盘图标 | [`tray-icon`](https://crates.io/crates/tray-icon)(Tauri 团队) | 成熟稳定,处理任务栏重启;或直接用 `Shell_NotifyIconW` |
| 绘制 | Direct2D(经 `windows` crate)+ GDI 兜底 | 灯的圆形、渐变、发光、动画 |
| HTTP/监听服务 | `tiny_http` 或 `std::net::TcpListener` 手写极简 HTTP | 仅需接收本地 POST JSON,无需重型框架 |
| JSON | `serde` + `serde_json` | 解析推送与配置 |
| 配置持久化 | `%APPDATA%\cc-status\config.json` | 免安装,用户级 |
| 开机自启 | 注册表 `HKCU\...\Run` 或启动文件夹 | 用户级,无需管理员权限 |
| 异步/并发 | 独立线程跑监听服务 + 自定义窗口消息通知 UI 线程 | UI 与网络解耦 |

> 说明:考虑过 `native-windows-gui`,但作者已声明其"功能完结、不再新增特性";对于本项目需要的自定义绘制与精确窗口定位,直接使用官方 `windows` crate 更灵活可控、更利于长期维护。

## 5. 状态与推送协议(已确认方向)

Claude CLI(Claude Code)支持 **Hooks**,可在会话生命周期的关键节点执行外部命令。**已确认**采用 Hooks 方案:由 hook 在关键事件(等待输入、请求授权、完成等)发起一次本地 `POST`,推送 JSON。cc-status 将随发布提供 hook 脚本模板。

```jsonc
POST http://127.0.0.1:9898/status
{
  "project": "my-repo",        // 项目标识(按此区分不同灯)
  "state":   "waiting_input",  // 状态枚举,决定灯的颜色
  "message": "需要你确认是否执行 rm -rf ...", // 悬停显示的消息
  "ts":      1752000000        // 时间戳(可选)
}
```

**项目生命周期(已确认)**:采用**心跳 + 超时自动移除**——每次推送即心跳;超过阈值 N 秒无推送 → 灯转灰色 `offline`;再持续更久 → 移除该灯。无需依赖显式退出事件。

**状态枚举与配色(已确认采用初步方案)**:

| 状态 | 含义 | 灯色 | 动画 |
|------|------|------|------|
| `idle` | 空闲/已完成 | 绿色 | 常亮 |
| `working` | 正在思考/执行 | 蓝色 | 呼吸 |
| `waiting_input` | 等待用户对话 | 黄色 | 闪烁/呼吸 |
| `waiting_permission` | 等待授权 | 橙色 | 闪烁 |
| `error` | 出错 | 红色 | 常亮/闪烁 |
| `offline` | 超时无心跳 | 灰色 | 常亮 |

## 6. 资源与依赖清单

- **构建**:Rust 1.88 stable + MSVC 工具链,目标 `x86_64-pc-windows-msvc`——**本机已就绪,原生构建,无需交叉编译**。
- **运行时**:Windows 10 1809+;无需 .NET、无需额外 VC++ 运行库(可静态链接 CRT)。
- **权限**:普通用户权限即可;开机自启用 `HKCU` 无需管理员。
- **crates**:`windows`、`tray-icon`、`serde`、`serde_json`、`tiny_http`(或手写)、`anyhow`、`log`/`tracing` 等。

## 7. 主要风险与对策

| 风险 | 等级 | 对策 |
|------|------|------|
| 托盘左侧定位在异形任务栏/第三方 shell 上失效 | 中 | 降级为右下角悬浮;托盘图标作为永远可用的入口 |
| DPI 缩放/多显示器导致错位 | 中 | 处理 `WM_DPICHANGED`,按每显示器 DPI 计算 |
| 置顶窗口遮挡其他 UI 或被全屏程序覆盖 | 低 | 全屏检测时自动隐藏;窗口极小、仅占灯区 |
| 端口被占用 | 低 | 启动时探测,失败则提示并允许改端口 |
| 本机开放端口的安全性 | 低 | 默认仅绑定 `127.0.0.1`;可选简单 token 校验 |
| Claude Code hook 能力/字段与设想不符 | 中 | 开发早期先在本机验证 hook 实际可发送的数据;协议保持宽松兼容 |

## 8. 结论

**项目在技术上完全可行,且开发环境即目标环境(本机 Windows + 完整 Rust 工具链),可原生构建与联调。** 所有需求均有成熟的 Win32/Rust 实现路径,唯一需要精心设计的"托盘左侧定位"已有明确方案与降级兜底。进入需求文档阶段,4 项关键决策已确认:①Hooks 推送;②心跳+超时移除;③初步状态配色方案;④本机原生构建。
